Git guide for kettles

Getting started

  • Installing Git

    I’ll assume you’ve installed Git already. Let’s not waste time and move to the next step.
    P.S. If you’re reading this, you’ve succesfully wasted your time. Bravo!

  • Creating a repository

    To create your first repository, open GitBash first. If you use Windows, it should be in Git folder in your program list (just open Windows menu). GitBash is a base and main util to interact with Git. However, it’s not required or even recommended to use a program whose interface ressembles old Linux OC. But, after all, it is working, so continule with it, if that’s your choice.

    Anyway, let’s complete at least this step. First of all, you’d want for Git to recognise you, which means it needs to know your name and, preferably, email. Use this:
    git config --global user.name "My Name"
    git config --global user.email myEmail@example.com
    And don’t forget to replace those examples with your actual email and a decent name - and if you’ll ever need to change it, just repeat one of these commands. Next step on our hard way… is already initialisation.
    First thing to do is find a project folder:
    cd C:/Users/user/Your_project
    Than just initialise it so Git knows it’s a repository:
    git init
    That’s it! Correct, it is indeed a repository and now you can store files in it as in a regular folder… do you? Well, Git doesn’t think so.

  • Adding in some files

    Well, while the files go in and out easily, Git doesn’t consider tham as something useable, and we should fix that. First, let’s see if Git found something for you. It’s simple:
    git status
    After that Git should tell you if it has tracked any files or just noticed something unidentified. In the last case, Git will need your permission on tracking them:
    git add example_text.txt
    It isn’t over yet. Git sees this file now, but doesn’t know what to do. This file, surely, exists, but to allow Git to freely manipulate it, we must commit… nothing, just commit:
    git commit -m 'The comment I left'
    Yeah, you leave a comment. Usually it’s whether of what the file is or what did you change in it. This procedure is required each time someone changes the file to. Now, available to commit files, it’s time to finally do something with your project.

  • Pushing and pulling

    What you need from now is to be able to save our project in a distant server and load it whenever you want, and that’s the easy part compared to what comes next. There are quite a lot of keywords, but for most of the time you’ll only need three, and those are either:
    git remote add repos_name <url>, it will add a distantly accesible repository by following url to your list of “remotes” or
    git push repos_name branch_name, this one pushes the required branch (“master” by default) to the repository or
    git pull, a fusion of git fetch and git merge commands in global size - it automatically fetches every single new or changed files from the version in repository and merges them with current version. Simply, it updates or recovers the project to the latest pushed version. And now, the most interesting part.

  • Branches

    This is quite a hard, but, indisputably, important feature that not only helps to keep multiple versions without saving them to the distant repository, but allows multiple people to work on the same project each implementing his own things. Branches, basically, are copies of the same project which can be changed separately and merged. And works this incredible tool like this:
    git branch name to create a branch with sertain name
    git checkout name to switch to this new branch or any other
    git merge name to try to merge current branch with the one with the name
    If the merge conflicts, Git will inform you and tell, where exactly the problems are. You’ll have to fix it manually. That’s it! Truly, those are the exact things you’ll need to operate this system. This isn’t everything, indeed, but a base is set and the work can begin!

  • Gitignore

    No, I didn’t lie, this was all. It’s just a quick tip to ease your work a bit. If you don’t want certain files, file types or folders to even be noticed by Git, you’ll need to follow several steps:

    • Create .gitignore text file in your root folder
    • Fill it with any names of files you don’t want to see Git showing you. You can use / before the name to mark it as a folder or * to mark it as file extension.

End
